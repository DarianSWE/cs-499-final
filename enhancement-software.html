Software Design & Engineering

A modular refactor and comprehensive input validation across a six-file C++ rebuild.

Outcome 1 · Collaborative Environments Outcome 5 · Security Mindset

Artifact: CoursePlanner-CS300.cpp · Originally from CS-300

The Artifact

This artifact is a C++ course planning application originally developed as the final project for CS-300: Data Structures and Algorithms. It loads course records from a CSV file into a custom hash table and provides a menu-driven interface for printing a sorted course list and searching for individual courses by ID. It began as a single, working file with no separation of concerns and no protection against invalid input.

Why This Enhancement

The original code had no real defense against failure. Invalid menu input caused a silent stream failure and an infinite loop. CSV rows were inserted without validation, so a malformed or empty line could corrupt the dataset silently. Print and search options could execute before any data was loaded, and nothing prevented a duplicate course ID from being inserted twice. None of these were edge cases — they were everyday user mistakes the original program had no way to handle gracefully.

What Changed

The single file was split into six dedicated files: Course.h for the data structure, HashMap.h/.cpp for the hash table, CourseManager.h/.cpp to encapsulate all application logic, and main.cpp for the menu loop alone. CourseManager owns an isLoaded flag that guards every operation requiring data, and every user-facing input path was hardened against failure.

// main.cpp — recovering from invalid menu input
if (!(cin >> choice)) {
    cin.clear();
    cin.ignore(numeric_limits<streamsize>::max(), '\n');
    cout << "Invalid input. Please enter a number." << endl;
    choice = 0;
    continue;
}

CSV rows are now validated before insertion — checked for a minimum field count, a non-empty course ID, and trimmed of stray whitespace — and the HashMap::insert() method rejects duplicate course IDs at the data layer rather than relying on the caller to check first.

Course Outcomes

The modular refactor demonstrates Outcome 1, building collaborative environments through maintainable code: separating logic into clearly scoped files with documented public interfaces is exactly the structure a team can divide and contribute to without conflict. The validation work demonstrates Outcome 5, a security mindset that anticipates failure: every fix here targets a real vulnerability — stream corruption, malformed input, premature data access, and duplicate insertion — rather than a hypothetical one.

Reflection

The most valuable part of this enhancement was discovering how much had been left unguarded in code I had once considered finished. Revisiting it with a critical eye revealed failure states I hadn’t thought to test and assumptions about input I hadn’t thought to question. Working through each of those gaps systematically clarified what separates functional code from code that’s actually ready to hand to someone else.
