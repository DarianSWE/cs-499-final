Algorithms & Data Structures

Replacing a collision-prone hash function with the industry-standard DJB2 algorithm.

Outcome 3 · Algorithmic Principles Outcome 4 · Innovative Techniques

Artifact: CoursePlanner-CS300.cpp · HashMap.h / HashMap.cpp

The Artifact

This enhancement builds on the same CoursePlanner application refactored in Category One, focusing specifically on the hash table at its core. The hash table uses separate chaining for collision resolution, each bucket is a vector of Course objects, so the performance of every insert and search depends directly on how evenly the hash function distributes course IDs across the table.

Why This Enhancement

The original hash function summed the ASCII values of every character in a course ID and took the result modulo the table size. That approach is not position-sensitive: course IDs that share a common prefix, like CS100 and CS200, differ by ASCII values one apart and hash to neighboring buckets. As the dataset grows, that clustering degrades average search time from O(1) toward O(n), undermining the entire reason a hash table is used in the first place.

What Changed

The summation hash was replaced with DJB2, a well-established algorithm developed by Daniel J. Bernstein. The bit-shift expression makes the result sensitive to character position, so structurally similar keys spread across the table rather than clustering together.

// HashMap.cpp — DJB2 hash function
unsigned long HashMap::hash(const string& key) const {
    unsigned long hashValue = 5381;
    for (char c : key) {
        hashValue = ((hashValue << 5) + hashValue) + c;
    }
    return hashValue % TABLE_SIZE;
}

The expression (hash << 5) + hash is equivalent to multiplying the running hash by 33 before adding each character, weighting each character’s contribution by its position. The seed value of 5381 is a prime chosen empirically for its distribution properties. The return type was also updated from int to unsigned long to safely accommodate the larger intermediate values produced during computation.

Course Outcomes

Outcome 3, designing and evaluating computing solutions using algorithmic principles, is demonstrated through the collision analysis of the original hash and the trade-off reasoning behind choosing DJB2. Outcome 4, applying well-founded and innovative techniques, is supported by the choice of an algorithm used in real production systems rather than an arbitrary fix.

Reflection

The most valuable part of this enhancement was developing a concrete understanding of why hash function design matters beyond producing a valid index. Seeing exactly how CS100 and CS200 land in adjacent buckets under summation made the performance argument for DJB2 tangible rather than theoretical, the kind of data-driven reasoning I expect to use regularly in analytics work.
